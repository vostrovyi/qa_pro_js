# Звіт про результати навантажувального тестування QA Pro REST App

## 1. Вхідні параметри та конфігурація середовища
* **Апаратне забезпечення:** Apple M4, 16Gb RAM.
* **ОС:** MacOS Tahoe 26.2.
* **Середовище:** Локальний запуск у Docker-контейнері (`localhost:3001`), сервер без підключеної зовнішньої бази даних.
* **Вимоги до продуктивності:** Latency (час відповіді) має складати **до 2 секунд**. 
* **Гіпотеза:** Якщо система "підвисає", це пов'язано з вичерпанням виділених ресурсів всередині Docker-контейнера, а не з апаратними обмеженнями хост-машини.

## 2. Цілі та параметри тестування
* **Ціль:** Оцінити потенційні можливості системи (Baseline) та знайти причину проблеми з продуктивністю при пікових навантаженнях (Stress).
* **Тип тесту:** Load Test (навантажувальне тестування з поступовим зростанням).
* **Інструмент:** Apache JMeter (використано Stepping Thread Group для контрольованого збільшення потоків).
* **Сценарій та Кореляція:** Створено реалістичний життєвий цикл сутності: `POST` (створення героя) -> `GET` (читання героя за згенерованим ID за допомогою техніки кореляції через **JSON Extractor**: `$.id`) -> `DELETE` (видалення цього ж героя).

## 3. Результати тестування та порівняльний аналіз
Для отримання об'єктивної картини було проведено два запуски: еталонний (150 паралельних запитів) та стресовий (1000 паралельних запитів). Моніторинг ресурсів проводився за допомогою вбудованої утиліти `docker stats`.

## 4. Інтерпретація результатів та виявлені проблеми
* **Джерело проблеми (Софт чи залізо?):** Обмеження є виключно програмними. Процесор Apple M4 працював у мінімальному режимі. Проблема криється в архітектурних лімітах Node.js або базової конфігурації контейнера, які не здатні обробити велику кількість одночасних TCP-з'єднань.
* **Поведінка користувачів:** Зависання сервера на термін понад 5 хвилин та 43% відмов призведуть до того, що користувачі не зможуть завершити жодної цільової дії (наприклад, оформлення замовлення) та покинуть ресурс.
* **Стан Docker-контейнера:** Падіння CPU до 0.00% при 1000 юзерах чітко вказує на стан **Deadlock**. Event Loop Node.js повністю заблокувався через переповнення черги запитів та вичерпання виділеного пулу пам'яті.

## 5. Стратегії оптимізації та рекомендації
1. **Горизонтальне масштабування:** Розгорнути декілька екземплярів додатку в Docker-мережі та налаштувати балансувальник навантаження (Nginx або HAProxy) для рівномірного розподілу запитів.
2. **Оптимізація Node.js та лімітів:** Збільшити об'єм виділеної оперативної пам'яті для контейнера та налаштувати прапорець `--max-old-space-size` для Node.js.
3. **Впровадження механізму захисту (Rate Limiting / Timeout):** Додати жорсткі таймаути на рівні серверного коду (наприклад, 5 секунд). Сервер має відкидати надлишкові запити з помилкою 429 (Too Many Requests), замість повного зависання системи та входу в Deadlock.
